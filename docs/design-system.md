# Steeefanie Personal Site 设计规范

> 版本：2.30
> 状态：当前项目设计基线
> 更新日期：2026-07-20
> 适用范围：主页、随笔、项目、食谱、长文详情页及后续轻量工具页面

## 1. 文档定位

本规范用于统一 Steeefanie Personal Site 的视觉语言、排版、布局、组件、交互、响应式和可访问性要求，是后续新增页面、调整样式和验收实现的依据。项目与食谱的公开路由、页面目录和内容集合统一使用复数 `projects`、`recipes`，不保留内部单数别名。

规范综合参考：

- Geist 的令牌化方法、排版尺度、4px 间距体系和克制的开发者界面风格；
- Apple Human Interface Guidelines 的层级、协调、一致性、适配、触控、动效和可访问性原则；
- 本项目“个人开发者主页 + 技术博客 + 项目案例”的实际信息架构；
- 当前 Astro、Markdown / MDX、三语言切换、搜索筛选和深浅主题实现。

本规范不要求模仿 Vercel 或 Apple 的品牌外观，不复制 Apple 系统控件、Liquid Glass 特效、SF Symbols 专有资源或页面结构。参考重点是设计方法和体验质量。

## 2. 设计目标

### 2.1 产品定位

网站首先是内容载体，其次才是个人品牌展示。核心目标是让访问者快速完成以下任务：

1. 了解 Steeefanie 的关注方向；
2. 浏览技术随笔和开发记录；
3. 查看项目背景、实现过程和使用方法；
4. 查阅菜谱、鸡尾酒配方和制作备注；
5. 在简体中文、台湾繁体、英文、深浅主题和不同设备间获得一致体验。

### 2.2 体验关键词

- 现代；
- 克制；
- 安静；
- 清晰；
- 内容优先；
- 技术感适度；
- 可长期维护。

### 2.3 核心原则

#### 内容优先

标题、摘要、正文和项目说明是视觉中心。装饰、插画、动效和容器不得压过内容。

#### 层级明确

通过字号、字重、间距、对齐和明度建立层级。颜色仅作为辅助，不单独承担状态表达。

#### 控制层与内容层分离

导航、语言切换和主题切换属于控制层，可以使用半透明和模糊；文章、项目和正文属于内容层，使用稳定背景，不使用大面积玻璃效果。

#### 一致而非完全相同

主页、随笔、项目和食谱共享基础视觉语言，但允许根据内容类型形成适当差异。随笔强调连续阅读，项目强调技术标签和实现过程，食谱强调材料、步骤、用时和份量。

#### 适配优先

布局必须适应屏幕宽度、字体缩放、语言长度和输入方式变化，而不是只适配某个设备截图。

#### 可访问性默认开启

键盘操作、焦点样式、减少动态效果、语义 HTML、合理对比度和足够大的触控区域属于基础要求，不是后续附加项。

## 3. 品牌与颜色

### 3.1 不可变品牌基准色

标准品牌色固定为：

```text
RGB: 99, 6, 95
HEX: #63065F
```

`#63065F` 是主品牌基准色，不得替换成 Apple 蓝、Geist 蓝或误写为 `#63055F`。仅允许在深色主题中使用本规范定义的可读性备用色 `#C979C4`，不得再派生第三种紫色。

品牌基准色主要用于：

- `Steeefanie` 站点名称；
- 重要品牌标记；
- 浅色背景上的链接和当前导航；
- 主要操作的实色背景；
- 浏览器顶部滚动进度条；
- 少量标签、代码强调和装饰细节。

### 3.2 主品牌色与深色备用品牌色

浅色主题使用主品牌色；深色主题使用为可读性设定的备用品牌色。每种主题内部只允许一个紫色色值：

```css
:root {
  --color-brand: #63065F;
  --color-brand-interactive: #63065F;
  --color-focus: #63065F;
}

:root[data-theme='dark'] {
  --color-brand: #C979C4;
  --color-brand-interactive: #C979C4;
  --color-focus: #C979C4;
}
```

使用规则：

- 浅色主题中，任何紫色文字、图标、下划线、焦点环和状态标记都严格使用 `#63065F`；
- 深色主题切换时，上述所有品牌与交互用途全局替换为备用色 `#C979C4`，不得混用主品牌色；
- 不再使用 `#B961B4`、`#D28DCE`、`#DC9AD7` 等第三种相近紫色；
- 需要弱背景时，只能以当前主题生效的品牌色增加透明度，不建立新的实色紫色；
- 大面积品牌实色按钮使用当前主题的 `--color-brand`，文字使用柔和的近白色；
- `#C979C4` 与深色页面、普通表面和抬升表面的对比度均达到普通文字WCAG AA底线；当前状态仍应同时使用下划线、字重、圆点或 `aria-current` 等非颜色信号。

品牌图标使用同一中心对称 S 形标志：浅色主题固定为 `#63065F`，深色主题固定为 `#C979C4`。标签页图标和导航字标只使用透明底版本，并跟随网站实际主题而不是仅跟随操作系统主题；带白色或深色圆角底板的版本作为后续应用图标资源保留，当前页面不引用。

### 3.3 浅色主题

浅色主题使用不带色相倾向的中性灰。页面底色为浅灰，内容表面和顶部导航可使用纯白；品牌紫色不得混入页面、表面、代码块或导航底色。

```css
:root {
  --color-page: #F5F5F5;
  --color-surface: #FFFFFF;
  --color-surface-raised: #ECECEC;
  --color-text-primary: #171717;
  --color-text-secondary: #666666;
  --color-border: #D9D9D9;
  --color-code-bg: #1F1F1F;
  --color-code-text: #F0F0F0;
  --color-header: rgba(255, 255, 255, 0.64);
  --color-focus: #63065F;
}
```

### 3.4 深色主题

深色主题同样使用无色相倾向的中性灰，不使用大面积纯黑。页面保持深灰，导航公共层比页面更深。

```css
:root[data-theme='dark'] {
  --color-page: #141414;
  --color-surface: #1D1D1D;
  --color-surface-raised: #262626;
  --color-text-primary: #F2F2F2;
  --color-text-secondary: #B3B3B3;
  --color-border: #333333;
  --color-code-bg: #0D0D0D;
  --color-code-text: #F0F0F0;
  --color-header: rgba(8, 8, 8, 0.64);
  --color-brand: #C979C4;
  --color-focus: #C979C4;
}
```

深浅主题不是简单反相。每个主题分别保证文字、边框、代码块和控制层具有稳定层次。

### 3.5 对比度基线

当前关键组合参考值：

| 前景与背景 | 对比度 | 用途判断 |
|---|---:|---|
| `#171717` / `#F5F5F5` | 16.44:1 | 浅色正文，充足 |
| `#666666` / `#F5F5F5` | 5.27:1 | 浅色次文字，满足普通文本 AA |
| `#63065F` / `#F5F5F5` | 11.19:1 | 浅色品牌文字，充足 |
| `#F2F2F2` / `#141414` | 16.46:1 | 深色正文，充足 |
| `#B3B3B3` / `#141414` | 8.79:1 | 深色次文字，充足 |
| `#C979C4` / `#141414` | 6.15:1 | 深色页面品牌文字，充足 |
| `#C979C4` / `#1D1D1D` | 5.63:1 | 深色普通表面品牌文字，满足 AA |
| `#C979C4` / `#262626` | 5.06:1 | 深色抬升表面品牌文字，满足 AA |

验收底线：

- 普通正文至少达到 WCAG AA 4.5:1；
- 大字号文字至少达到 3:1；
- 关键正文和长时间阅读内容优先接近或达到 7:1；
- 焦点、当前状态和错误信息不能只通过颜色表达。

### 3.6 状态颜色

当前站点以阅读为主，不建立大量彩色状态体系。项目不区分“已发布、实验、进行中”等状态，也不显示状态圆点；导航当前项、语言选择、筛选选中等交互状态必须同时依靠语义属性或形态变化表达，不能只依靠颜色。

如未来需要成功、警告和错误颜色，应新增语义令牌，不直接在组件中写颜色值，并分别验证深浅主题对比度。

## 4. 字体与排版

### 4.1 字体栈

```css
--font-sans-en: "Inter", -apple-system, BlinkMacSystemFont, "Segoe UI", Arial, sans-serif;
--font-sans-zh: "Noto Sans SC", "Source Han Sans SC", "PingFang SC", "Microsoft YaHei", sans-serif;
--font-mono: "JetBrains Mono", "SFMono-Regular", Consolas, monospace;
```

规则：

- 中文和英文正文均使用无衬线字体；
- 不因参考 Geist 而强制换用 Geist Sans；
- 不因参考 Apple 而复制或依赖 SF Pro；
- 每种语言只使用一套主字体栈；
- 代码和等宽数据使用 `--font-mono`；
- 标签的拉丁字符优先使用等宽字体；简体中文标签必须在通用 `monospace` 之前回退到 `Noto Sans SC / Source Han Sans SC / PingFang SC / Microsoft YaHei`，台湾繁体对应回退到 `Noto Sans TC / Source Han Sans TC / PingFang TC / Microsoft JhengHei`，避免 Windows 因 Consolas 缺少中文字形而落入宋体；
- 在线字体加载失败时必须有可靠系统回退。

### 4.2 字重

全站原则上只使用三档字重：

```text
400 Regular：正文、摘要、说明
500 Medium：导航、标签、次级标题、按钮
600 Semibold：页面标题、章节标题、重点内容
```

避免：

- 520、560、590、650 等依赖可变字体的零散字重；
- 300 及以下的细字重；
- 大面积 700 或 800 粗体；
- 仅通过加粗区分所有层级。

### 4.3 排版令牌

| 令牌 | 桌面尺寸/行高 | 字重 | 典型用途 |
|---|---|---:|---|
| `display` | `clamp(52px, calc(40px + 3vw), 80px)` / 1.06 | 600 | 首页主标题 |
| `page-title` | `clamp(44px, 6vw, 72px)` / 1.08 | 600 | Blog、Project 页标题 |
| `section-title` | `clamp(24.8px, 3vw, 35.2px)` | 600 | 首页“最近更新”和所有文档详情标题，共用 `--font-size-section-title` |
| `article-title` | `var(--font-size-section-title)` / 1.25 | 600 | 随笔、项目和食谱详情标题，不得在单篇内容中覆盖 |
| `heading-2` | 32px / 40px | 600 | 页面主要章节 |
| `heading-3` | 24px / 32px | 600 | 长文三级标题 |
| `heading-4` | 18px / 28px | 600 | 长文四级标题 |
| `body-large` | 18px / 30px | 400 | 首页介绍、页面导语 |
| `body` | 16px / 28px | 400 | 普通正文 |
| `body-zh` | 16px / 29px | 400 | 中文长文正文 |
| `label` | 14px / 20px | 500 | 导航、按钮、标签 |
| `meta` | 13px / 18px | 400 | 日期、阅读时间、状态 |
| `code` | 14px / 22px | 400 | 代码块 |

大标题字距允许收紧，正文不使用负字距：

```text
display（英文）：-0.04em
display（中文混排）：-0.025em
page-title：-0.04em
section-title / article-title：-0.035em
heading-2：-0.025em
正文与元信息：normal
```

负字距不是装饰效果，也不应普遍用于正文。大号拉丁字母的字偶间距和字形两侧留白会随字号同步放大，保持默认字距时容易显得松散，因此展示标题允许轻微收紧，以恢复词形整体性和稳定的视觉密度。中文方块字本身占据较完整的字面空间，只需更弱的负字距；中英文混排不得直接套用强拉丁字距，否则中文笔画和标点会显得拥挤。字距必须随字号和文字系统分别校准，正文、摘要与元信息保持正常字距。

简体中文、台湾繁体和英文的长正文顶层段落统一使用固定 `text-indent: 2rem`。不得使用随局部字号变化的 `2em`，避免相同正文规则在不同字号环境中产生不同的物理缩进距离。缩进只用于随笔、项目和食谱详情正文，不用于页面导语、摘要、卡片、列表、标题、引用、提示块、图注、按钮和其他界面文案；表格及其单元格始终取消缩进。

主页介绍、“关于我”正文和随笔、项目、食谱详情页中的正文段落均使用两端对齐，末行保持左对齐。中文使用字符间调整，英文使用词间调整并按当前 `lang` 启用自动断词；标题、摘要、列表说明、元信息、按钮和代码块不使用两端对齐。

### 4.4 阅读宽度

- 全站宽容器：1120px；
- 文章正文：680–740px；
- 中文正文优先保持较短行长；
- 英文正文建议每行约 60–80 个字符；
- 代码块和表格可在正文容器内部横向滚动，不扩大页面宽度。

### 4.5 文本缩放

布局必须在以下情况下保持可读：

- 浏览器缩放 200%；
- 320px 窄屏；
- 中英文长标题；
- 系统较大字体；
- 日期和标签长度增加。

重要内容不得通过固定高度裁切。导航和控件不得因文字变长而互相覆盖。

## 5. 间距体系

### 5.1 基础尺度

采用 4px 基础单位：

```text
space-1   4px
space-2   8px
space-3  12px
space-4  16px
space-6  24px
space-8  32px
space-10 40px
space-12 48px
space-16 64px
space-20 80px
space-24 96px
```

所有新增间距原则上从该尺度选择。确有排版需要时可以使用其他 4px 倍数，但不新增 17px、27px、42px、54px 等缺少体系依据的值。

### 5.2 使用节奏

- 4–8px：图标和文字、同组元信息；
- 12–16px：标题与摘要、相邻控制项；
- 24–32px：列表项内部、内容小组之间；
- 40–48px：标题区与列表、章节内部；
- 64–96px：首页大区块和页面主章节。

间距优先用于表达关系：越相关越接近，越独立越分开。

## 6. 布局与响应式

### 6.1 容器

```css
--layout-wide: 1120px;
--layout-reading: 740px;
```

建议页面内边距：

```text
桌面：20px 以上
平板：20px
手机：14–16px
```

### 6.2 项目断点

为保持与现有实现一致，使用以下行为断点：

```text
Compact：≤ 720px
Medium：721–800px
Wide：> 800px
```

断点用于内容需要发生变化的位置，不按具体设备型号增加断点。

验收视口至少包括：

```text
320×568
390×844
768×1024
1024×768
1440×900
```

### 6.3 对齐

- 导航内容与页面宽容器使用同一左右基准；
- 正文详情页使用独立阅读容器；
- 文章和项目列表的标题、摘要、元信息保持稳定列关系；
- 移动端改为单列，元信息左对齐；
- 不使用无意义的居中对齐长正文；
- 三语言切换后整体基线和导航宽度保持稳定；四个导航项与搜索、语言、主题三个图标按钮使用同一44px控制高度。三个图标允许在命中区域不变的前提下视觉下移1px，以补偿图标与文字字面的光学中心差异。
- `Steeefanie` 是固定拉丁字标，其中首字母 `S` 由透明底品牌 SVG 替代，后续文字固定为 `teeefanie`。标志使用11×16px可见裁切框、18×18px居中图像和1px光学间距；完整链接的无障碍名称仍为 `Steeefanie`。文字统一使用拉丁字体栈、16px字号、600字重和1倍行高，不继承中文语言字体栈，也不在移动断点单独缩放。

### 6.4 首页布局

首页顺序固定为：

```text
欢迎语与个人简介
最近更新（随笔、项目、食谱各取最新1条，共3条，按发布日期重新排序）
页脚
```

- 欢迎语上方不设置英文眉题或“开发 · 记录 · 分享”等口号；
- 首页说明分别自然引导至随笔、项目和食谱，中文使用“技术笔记、踩坑经历、碎碎念”“做过的菜、揉过的面、调过的酒”等生活化表达；英文采用自然意译，并使用 `dough I’ve kneaded` 表达发面、馒头和发糕一类面食语境，不写作 `noodles`；
- “关于我”正文不使用首行缩进；
- “关于我”使用独立的中性抬升表面，其下方设置联系方式行：左侧显示纯文本 `Contact: taichieh3@gmail.com`，其左边线距卡片左边12px，与12px圆角曲线和底边直线的交界点对齐；右侧GitHub图标保留外部链接和明确的无障碍名称；
- “最近更新”标题下不再附加“各取最新一条”等机制说明，具体选择逻辑只在数据层和维护文档中保留。
- “最近更新”桌面列表中的日期顶端与条目标题顶端对齐，不与栏目标签顶端对齐；移动端日期随元信息自然排在正文下方。
- 页脚只保留动态年份版权标识 `© 年份 Steeefanie`，不附加框架宣传或泛化口号。

不增加：

- 独立 About 导航；
- Skills 技能墙；
- Testimonials；
- Clients；
- Newsletter 营销模块；
- 数据指标和虚构业绩。

### 6.5 随笔布局

- 列表页不显示大字号栏目标题、英文眉题或栏目说明，进入主内容后直接展示列表；SEO标题与页面描述继续保留在文档元数据中；
- 列表页不提供标签筛选；
- 使用纵向连续列表；
- 标题是视觉核心；
- 摘要控制在约 2–3 行自然阅读长度；
- 日期、阅读时间和标签弱化；
- 条目通过留白、标题层级和轻微交互背景区分，不绘制常驻灰色分割线；
- 不使用密集卡片网格和大面积封面图。

### 6.6 项目布局

- 列表页不显示大字号栏目标题、英文眉题或栏目说明，进入主内容后直接展示列表；SEO标题与页面描述继续保留在文档元数据中；
- 列表页不提供状态或技术栈筛选；项目内容模型不设置状态字段，列表与详情均不展示状态；
- 列表强调项目名称、说明、更新时间和技术标签；
- 由生成式人工智能辅助整理的项目可使用可选 `attribution` 字段，在摘要前以弱化的小字号文字标注，不使用徽章、边框或背景块；
- 每个项目拥有独立详情页；
- 项目填写 `repositoryUrl` 后，在详情标题右侧显示单色 GitHub 图标并链接到对应仓库；图标默认使用次要文字色，hover 使用当前主题交互色，不设置背景块，并提供三语言无障碍名称；
- 详情页允许长文、代码、表格、图片和外部链接；
- 同组过程图或结果图在桌面端可使用两列图廊，720px及以下改为单列；每张图片必须可打开原图，图注不缩进；
- 性能或算法测试文章按“用途—处理思路—测试设置—实测结果—输出对比—使用与边界”的顺序组织；颜色数量、压缩率、色差和耗时必须来自可复现测试，不以示例数字代替实测值；原图与每种被比较模式的输出均需展示，测试范围和计时边界必须写清；
- 项目描述不能退化为只有几行的博客摘要；

### 6.7 食谱布局

- 列表页不显示大字号栏目标题、英文眉题或栏目说明，进入主内容后直接展示类别筛选和列表；SEO标题与页面描述继续保留在文档元数据中；
- 列表仅保留“全部、家常菜、面食、鸡尾酒”类别筛选，不提供标签筛选；
- 食谱列表同时容纳菜谱和鸡尾酒配方；
- 每条食谱包含名称、简介、类别、制作时间、份量和标签；
- 食谱列表按 frontmatter 中代表实际记录时间的 `publishDate` 倒序排列，不读取上传时间、文件修改时间或构建时间；同日条目保持稳定内容顺序；
- 同一基础配方仅因液体比例、口感目标等少量参数形成的多个版本，应合并在同一篇食谱中，以普通材料列表和清晰的小节区分，不拆成重复列表项；只有多个字段必须逐项比较时才使用表格；
- 家常菜与面食的做法使用 Markdown 有序列表，页面必须恢复并显示十进制序号，不得因 CSS Reset 隐藏列表标记；
- 每个鸡尾酒条目必须为材料中实际使用的全部基酒分别添加稳定标签，不得只写“混合基酒”等概括标签；简体中文的 `gin` 固定显示为“金酒”，台湾繁体显示为“琴酒”，英文显示为 `Gin`。朗姆酒基底利口酒按其明确基酒归类，只有“某基酒风味”但未确认实际含有该基酒的预调酒不得推断添加；
- `cocktail` 只作为类别枚举使用，不再重复添加为可见标签；鸡尾酒条目不显示泛化的 `#鸡尾酒 / #調酒 / #Cocktail` 标签；
- 详情正文至少包含材料和做法，可增加备注、调整建议和器具说明；
- 鸡尾酒做法只写与配方结果直接相关的操作，不指定高球杯、古典杯等盛装杯型；需要描述容器时统一使用“杯中／glass”等中性表述，雪克壶、滤纸等实际操作工具不受此限制；
- 鸡尾酒制作时间按完成一杯的实际操作时长标注：直接调和类通常约2分钟，摇和或分层类通常约3分钟，需要双重过滤的奶洗类按实际过滤过程单独估算；不把不存在的静置步骤或宽松预留时间计入；
- 类别值在数据层使用 `home-cooking`、`flour` 和 `cocktail`，简体中文显示“家常菜 / 面食 / 鸡尾酒”，台湾繁体显示“家常菜 / 麵食 / 調酒”，英文显示“Home cooking / Flour dishes / Cocktail”；
- 鸡尾酒内容应包含适度饮酒和法定饮酒年龄提示；
- 示例食谱必须明确可替换，不虚构为个人原创或经过长期测试的食谱；
- 食谱仍以文字排版为核心，不制作图片主导的卡片墙。

## 8. 导航与控制层

### 8.1 顶部导航

导航顶部不设置阅读进度轨道。导航背景从视口顶边开始，所有品牌文字、主导航文字和工具图标在共同居中基线的基础上整体向上微调1px；不得针对单个语言另设垂直偏移。

桌面结构：

```text
简中：Steeefanie      主页  随笔  项目  食谱  搜索  主题  语言
繁中：Steeefanie      首頁  隨筆  專案  食譜  搜尋  主題  語言
英文：Steeefanie      Home  Blog  Projects  Recipes  Search  Theme  Language
```

要求：

- 导航横跨浏览器宽度；
- 内部与页面宽容器对齐；
- 桌面高度 72px，移动端主行高度 64px；
- 站点名称始终可点击返回首页；
- 当前路由使用颜色和下划线共同表达；
- 非当前项保持清晰但不过度突出；
- 不使用厚重按钮式导航。
- 导航简体中文名称固定为“主页、随笔、项目、食谱”，台湾繁体固定为“首頁、隨筆、專案、食譜”；
- 导航英文名称固定为“Home、Blog、Projects、Recipes”。
- 桌面工具按钮顺序固定为搜索、主题、语言；720px及以下在语言按钮右侧增加两横线菜单按钮；
- 720px及以下隐藏行内导航，四个导航项只在导航下方的全宽临时面板中显示，不新增永久第二行；
- 移动导航文字统一靠右排列，手机端文字右边缘距视口约45px；各导航项之间不绘制横向分割线；
- 精细指针设备支持悬停和键盘聚焦展开，移出150ms后关闭；触屏设备通过点击切换；
- 搜索、语言和移动导航面板互斥，均支持外部点击、Escape和选择后关闭。
- 面板打开时，公共材质层和文字安全遮罩均使用320ms并同时结束；文字不得在材质层覆盖前突然出现，也不得在材质层停止后继续缓慢补显；
- 材质层移动期间，其下边沿以内96px（约两个移动导航文字行高）必须保持完全无文字；再向上设置80px透明羽化区，由完全透明逐渐过渡到完整文字；
- 文字安全遮罩在最后约38ms与材质层同步完成最后一段移动，不得出现材质已停止但文字仍在补显的尾段；
- 每行文字使用200ms独立透明度动画，相邻行延迟22ms并相互重叠；不得使用整块文字上下位移，也不得把行间空白计入显现进度；
- 面板关闭必须执行反向逻辑：先用48ms建立96px无文字安全区和80px羽化，再同时启动文字遮罩与材质层的320ms上收；二者必须使用相同的 `cubic-bezier(.2,.72,.2,1)` 缓动和位移比例，使文字透明边界在整个收起过程中始终位于材质边界上方至少96px。各行文字以180ms反向交叠淡出，底部项目先退场，不得直接隐藏文字或让文字越过材质边沿。

### 8.2 材质与毛玻璃

半透明模糊只用于顶部公共材质层和临时控制层：

```css
backdrop-filter: blur(20px);
```

规则：

- [Apple HIG Materials](https://developer.apple.com/design/human-interface-guidelines/materials) 使用 `ultraThin / thin / regular / thick` 等语义材质而非固定网页像素区间；本站导航文字量介于 thin 与 regular 的使用场景之间，采用偏清透配置；
- 参考 [Apple WWDC25「What’s new for the spatial web」](https://developer.apple.com/videos/play/wwdc2025/237/)示例采用的 `blur(20px)` 以及0.8至0.5半透明底色，本站保持 `blur(20px)`，将底色透明度取为偏清透的0.64；
- 浅色公共层以纯白为色源，使用 `rgba(255, 255, 255, 0.64)`，合成后比 `#F5F5F5` 页面更亮；深色公共层使用 `rgba(8, 8, 8, 0.64)`，比 `#141414` 页面更深；
- 公共层不提高透过内容的饱和度，避免品牌紫色污染中性底色；
- 页面、内容表面、抬升表面、代码块、灰色边界和导航底色都必须使用 R=G=B 的中性灰；品牌紫色只用于品牌与交互语义；
- 背景必须有足够不透明度保证文字可读；
- 3px进度轨道、导航和展开面板由同一个公共材质层承载，导航自身保持透明；
- 进度条完成区域显示品牌色，未完成区域直接露出公共毛玻璃材质；
- 进度条与导航之间、公共材质层底部均不绘制分割线；
- 导航隐藏时公共材质层收缩到3px，搜索或菜单面板打开时强制导航可见；
- 不添加明显投影；
- 不把 Liquid Glass 扩散到文章、项目列表或正文；
- 不实现高成本折射、流体高光和持续跟随效果。

### 8.3 滚动行为

- 向下滚动超过阈值后导航平滑隐藏；
- 向上滚动超过阈值后导航重新显示；
- 页面接近顶部时导航始终显示；
- 微小滚动不反复触发；
- 使用 `transform`，避免频繁触发布局；
- 顶部阅读进度条在导航隐藏时继续可见；
- 减少动态效果时缩短或移除位移动画。
- 搜索和菜单展开时不使用遮罩，也不降低页面亮度。

### 8.4 搜索面板

- 搜索按钮位于语言按钮左侧，面板从导航下方全宽展开；
- 打开搜索后，顶部公共材质层必须向下覆盖完整 `100dvh`；搜索面板占据导航以下的剩余动态视口并独立滚动，新增结果不得落到无材质背景上；
- 搜索打开期间锁定正文在进入搜索前的滚动位置；滚轮、触控滑动和惯性滚动只交给搜索面板处理，关闭时临时绕过全局平滑滚动并立即恢复正文位置，不允许滚动穿透或产生二次回滚动画；
- 默认搜索范围跟随当前路由：首页为“全部”，随笔为“随笔”，项目为“项目”，食谱为“食谱”；每次重新打开搜索时都恢复该页面对应的默认范围；
- 搜索范围为标题、摘要、标签和技术栈等元数据，不索引 Markdown 正文；
- 搜索索引在构建时输出为 `/search-index.json`，不依赖第三方搜索服务；
- 查询先进行 NFKC 归一化，英文匹配不区分大小写；
- 标题匹配权重高于摘要，摘要高于标签与技术栈；显示全部匹配结果和结果数，不设8条上限；结果超出视口时由搜索层独立滚动；
- 搜索输入框提示为简体中文“搜索标题、摘要或标签”、台湾繁体“搜尋標題、摘要或標籤”和英文“Search titles, summaries, or tags”；技术栈仍按标签类关键词参与搜索；
- 范围筛选必须随语言同步：简体中文“全部、随笔、项目、食谱”，台湾繁体“全部、隨筆、專案、食譜”，英文“All、Blog、Projects、Recipes”；
- 搜索范围的 hover 和选中态沿用导航栏样式：保持透明、取消圆角矩形背景和描边，使用主题色文字与1.5px底部短线；键盘 `focus-visible` 焦点环继续保留；
- 输入为空时不显示预搜索占位状态，也不显示键盘操作说明；
- 搜索输入区不绘制圆角矩形背景、边框或阴影，输入框与面板公共材质直接融合；
- 支持方向键选择、Enter打开、Escape关闭，并使用可感知的活动项状态；
- 面板不得添加页面遮罩、背景压暗或内容缩放。

## 9. 控件与交互状态

### 9.1 命中区域

所有移动端常用按钮实际命中区域至少为：

```text
44×44px
```

图标视觉尺寸可以保持 18–20px，但外部按钮区域不得缩小到 34px。桌面端也应保持足够的键盘焦点和鼠标命中空间。

### 9.2 必备状态

每个按钮和链接至少实现：

```text
default
hover
focus-visible
active / pressed
disabled（如适用）
current / selected（如适用）
```

状态规则：

- hover 只能作为辅助，功能不能依赖 hover；
- active 必须有即时反馈；桌面和iPad的单行列表可使用轻微位移，720px及以下的多行列表不得改变横向内边距、缩放或宽度，只通过严格的主题色变化反馈，避免触摸后重新换行；
- focus-visible 必须清晰可见；
- disabled 不响应操作，并提供视觉和语义状态；
- 桌面持久导航的当前页面不能只改变颜色，应增加下划线；临时移动菜单允许使用主题交互色配合 `aria-current` 表示当前页面。
- 搜索、语言、主题和菜单图标按钮悬停时保持透明背景，只将图标变为主题交互色；

### 9.3 焦点样式

优先使用两层焦点环，保证在复杂背景上仍可识别：

```css
:focus-visible {
  outline: none;
  box-shadow:
    0 0 0 2px var(--color-page),
    0 0 0 4px var(--color-focus);
}
```

无法使用 `box-shadow` 的文本链接可采用 2px outline 和 3–4px offset。

## 11. 主题切换

### 11.1 控件

- 使用太阳短射线与右下月相切口组合图标，形态参考用户提供的明暗主题图并重绘为代码内SVG；
- 月相主体不得添加完整圆形描边；由右下方月牙的弧线、人眼视觉补齐和外围射线共同建立圆形感，月相位置与盈亏程度以当前第一版图标为基准；
- 图标使用 `currentColor`，不得嵌入固定黑白位图或依赖主题背景色切口；
- 图标按钮具有动态 `aria-label`；
- 图标视觉尺寸约 18–20px；
- 按钮命中区域至少 44×44px。

### 11.2 初始化优先级

```text
1. localStorage 已保存主题
2. prefers-color-scheme
3. 浅色主题
```

主题初始化应在页面绘制前尽早执行，避免闪白或闪黑。

### 11.3 扩散动画

- 切换开始时必须完整保留旧主题，不得先将整个页面立即切换到新主题；
- 浏览器先捕获旧主题，再提交新主题状态；
- 新主题作为新视图，从当前可见主题图标的几何圆心以圆形裁剪区域逐渐显现；
- 圆扩散到哪里，哪里才显示新主题；圆外区域在动画完成前继续显示旧主题；
- 圆心必须读取 SVG 图标的实时边界中心，不使用整个 44px 按钮命中区域的中心代替；
- SVG 中心坐标必须换算为根视口宽高内的百分比，再传给圆形裁剪；不得直接把页面 CSS 像素作为快照伪元素的局部坐标，以免桌面快照发生缩放时圆心偏移；
- 半径覆盖最远视口角点；
- 覆盖半径与圆心百分比基于同一冻结根快照视口；宽度必须使用包含经典滚动条区域的 `window.innerWidth`，不得使用会少一个滚动条宽度的 `documentElement.clientWidth`；快照组固定宽高且 `transform: none`；
- 根元素使用 `scrollbar-gutter: stable` 稳定保留经典滚动条沟槽，避免 Windows Chrome 在快照层与真实页面之间出现横向抖动；
- 旧主题快照固定 `opacity: 1`、`filter: none`，不得参与淡出、压暗、缩放或透明度动画；仅裁剪新主题快照；
- 视觉节奏按半径匀速，使用 `r(t)=Rt` 的线性关键帧；时间25%、50%、75%时半径分别达到覆盖半径的25%、50%、75%；
- 总时长按覆盖半径计算为 `clamp(280ms, radius / 3.3, 440ms)`，手机保持快速，大屏增加有效帧数；
- 圆形裁剪动画必须预先声明在 `::view-transition-new(root)` 上，使其在新视图图层生成时同步启动；
- 不得先将 View Transition 动画全部置空，再等待 `transition.ready` 后通过 JavaScript补挂动画，以免首次加载时产生伪元素生命周期竞态；
- 根快照组允许关闭默认几何插值，但必须明确冻结其视口宽高和变换，不能让浏览器在新旧几何之间缩放；
- 优先使用 View Transitions API 保证背景、文字、边框和控件同步切换；
- 不支持 View Transitions 时降级为短暂淡出、提交主题、再淡入，不允许出现突兀闪白或闪黑；
- 不引发布局跳动；
- 快速连续点击必须加锁或合并；
- 不支持高级动画时降级为短暂渐变；
- 主题图标在扩散期间保持位置和姿态稳定，不叠加旋转动画；
- `prefers-reduced-motion: reduce` 时取消扩散。

该动画是主题状态变化反馈的特例，不作为其他页面大范围使用动画的依据。

## 12. 动效规范

### 12.1 时间令牌

```css
--motion-instant: 100ms;
--motion-fast: 150ms;
--motion-standard: 200ms;
--motion-overlay: 300ms;
--motion-theme: 360ms; /* 动态时长的中间参考值 */
```

### 12.2 使用范围

| 场景 | 建议时长 |
|---|---:|
| 链接颜色、按钮背景 | 100–150ms |
| 按下反馈 | 100–140ms |
| 导航下划线 | 150–180ms |
| 导航显隐 | 200–220ms |
| 普通浮层 | 200–300ms |
| 主题圆形扩散 | 280–440ms，半径线性 |

### 12.3 原则

- 动效用于说明状态、层级或操作结果；
- 不添加持续循环和无意义背景动画；
- 不使用视差、大面积漂移和强烈弹跳；
- 不依赖动画传递唯一重要信息；
- 减少动态效果时优先使用颜色变化、淡入淡出或直接切换；
- 滚动监听使用被动监听和 `requestAnimationFrame`。

## 13. 列表与内容组件

### 13.0 列表筛选

- Blog 与 Projects 不提供列表筛选；Recipes 仅提供家常菜/面食/鸡尾酒类别单选筛选；
- 筛选值使用稳定类别枚举，不使用本地化显示文字作为 URL 值；
- 食谱类别筛选的 hover 和选中态使用主题色文字与1.5px底部短线，不使用胶囊、圆角矩形背景或选中底色；键盘焦点环不受影响；
- 筛选状态写入 URL 查询参数，并在刷新、前进后退和分享链接后恢复；
- 无结果时按当前语言显示空状态和“清除筛选”；
- 食谱筛选只显示“全部、家常菜、面食、鸡尾酒”四个选项，不显示“类别”等可见标题；筛选组仍保留仅供辅助技术读取的隐藏图例；四个选项的字号与同页文章摘要一致，桌面、iPad和手机均使用1rem并始终保持一行；窄屏只压缩间距与水平内边距，不缩小文字，不得换行或引起页面级横向滚动。

### 13.1 列表分页

- Blog、Projects 和 Recipes 列表每页最多显示20条，超出后显示上一页、页码和下一页操作；
- 页码写入 `?page=N`，第1页不保留冗余 `page=1`；刷新、前进后退和分享链接后应恢复当前页；
- Recipes 先按类别筛选，再对筛选结果分页；更换类别或清除筛选时回到第1页，筛选参数与页码参数可同时存在；
- 内容不足20条或筛选无结果时隐藏分页控件；越界、非数字或非正整数页码自动归一到有效页；
- 分页不使用胶囊、圆角矩形背景或分割线；当前页使用主题色文字和底部短线，方向按钮在边界页显示禁用状态。

### 13.2 博客列表项

必须包含：

- 标题；
- 摘要；
- 发布日期；
- 阅读时间；
- 标签。

样式要求：

- 标题为主要视觉元素；
- 摘要使用次文字色；
- 元信息使用 `meta` 令牌；
- 上下使用留白形成节奏，不绘制常驻灰色分割线；
- hover 可轻微水平位移或改变标题颜色；
- 不增加厚重卡片背景和明显阴影。

### 13.3 项目列表项

必须包含：

- 项目名称；
- 一句话说明；
- 更新时间；
- 技术栈。

项目列表和详情的本地化日期均显示年、月、日，不省略日期中的“日”。

### 13.4 标签

- 使用等宽字体或较小无衬线字体；
- 普通标签不制作大胶囊；
- 标签之间保持 8–12px 间距；
- 标签颜色不得压过标题；
- 所有标签统一添加 `#` 前缀；技术名称按独立标签展示，例如 `#Astro #TypeScript #Tailwind #CSS`，不合并为 `#Tailwind CSS`；
- 标签文本本身不添加首尾空格；多语言标签容器和普通标签容器统一使用 CSS `gap` 产生间距，避免文本空格与布局间距叠加；
- 标签过多时自然换行，不横向撑破页面。

## 14. 长文排版

### 14.1 结构

Blog 详情页顺序：

```text
返回入口
标题
摘要
日期 / 更新时间 / 标签 / 阅读时间
正文
上一篇 / 下一篇
```

详情标题必须读取内容 frontmatter 的 `title` 并由统一模板输出；Markdown / MDX 正文不得重复书写一级标题，也不得在单篇内容中覆盖标题字号。标题字号与首页“最近更新”共用 `--font-size-section-title`，因此替换示例文档或新增内容时无需重复配置。

详情元信息统一字号与行高，并在同一行盒中垂直居中。日期、阅读时间、类别和标签不得各自使用不同字号或嵌套基线规则；类别圆点随文字行盒居中。上一篇/下一篇使用显式左右插槽：上一篇固定左列，下一篇固定右列；缺少任一方向时不渲染空占位，720px及以下统一为单列左对齐。

详情页头统一使用紧凑节奏：返回入口至标题16px，标题至摘要16px，摘要至元信息20px；页头下边距和正文上边距各32px。正文每种语言内容的第一个标题不再叠加通用章节上边距，使元信息到正文首标题的视觉间隔稳定为64px。该规则由公共样式控制，单篇Markdown不得自行插入空段落补偿。

Project 详情页顺序：

```text
返回入口
项目名称
一句话说明
时间 / 技术标签
正文
可选 Demo / Repository / Download
```

Recipe 详情页顺序：

```text
返回入口
食谱名称
一句话说明
菜谱/鸡尾酒类别 / 制作时间 / 份量 / 标签
材料
做法
备注或调整建议
上一篇 / 下一篇
```

### 14.2 正文

- 中文正文行高约 1.8；
- 英文正文行高约 1.7–1.75；
- 中英文正文段落均两端对齐且末行左对齐；英文启用自动断词，避免过大的词间空白；
- 三种语言的顶层正文段落统一使用固定 `2rem` 首行缩进；列表、引用、图注、表格和表格单元格不继承正文缩进；
- 有序列表必须显式使用十进制序号，无序列表必须显式恢复圆点，不能依赖浏览器在 CSS Reset 之前的默认样式；
- 标题层级必须连续、清晰；
- 段落之间通过间距区分；
- 链接使用品牌色和下划线；
- 引用使用细品牌色左边框；
- 表格在正文容器内水平居中，表格单元格文字居中且无首行缩进；超出阅读宽度时只在表格容器内部横向滚动；
- 图片必须有 `alt`；
- 大图避免阻塞首屏；正文图廊中的非首屏图片使用懒加载。

### 14.3 代码

- 使用等宽字体；
- 背景与正文形成清晰但柔和的层次；
- 横向溢出时只滚动代码块；
- 不使用过度花哨的语法主题；
- 行内代码与代码块有明确区别；
- 代码不得包含真实密钥、Token、密码或个人敏感路径。

## 15. 图片、图标与资源

### 15.1 图片

- 首页不使用无意义大幅插画；
- 项目截图必须服务于功能说明；
- 优先使用 WebP、AVIF 或合理压缩的 PNG；
- 明确宽高，减少布局跳动；
- 为有信息价值的图片提供准确替代文本；
- 装饰图片使用空 `alt`；
- GIF 需要控制体积，并考虑减少动态效果场景。

### 15.2 图标

- 优先使用简洁 SVG；
- 视觉语言保持一致；
- 不直接复制受限制的 Apple 专有图标资产；
- 图标按钮必须有 `aria-label`；
- 重要信息不能只依靠图标表达。

## 16. 无障碍规范

必须满足：

- 使用语义化 HTML；
- 顶部导航使用 `<nav>`；
- 主内容使用 `<main>`；
- 每页只有一个主要 `<h1>`；
- 标题层级不无故跳跃；
- 所有功能支持键盘访问；
- 有清晰 `focus-visible`；
- 图标按钮具有可理解的动态标签；
- 图片具有合适 `alt`；
- 颜色不是唯一状态提示；
- 控件命中区域至少 44×44px；
- 支持 `prefers-reduced-motion`；
- 支持 `prefers-color-scheme`；
- 语言切换更新 HTML 语言属性；
- 外部链接在上下文中能够被识别；
- 页面缩放 200% 后仍可使用；
- 手机端无页面级横向滚动。

## 17. 内容语言与文案

### 17.1 中文

- 清楚、自然、直接；
- 不使用企业营销式口号；
- 不虚构用户量、收入、客户和性能指标；
- 技术内容说明背景、边界、实现和限制；
- 标题准确表达文章主题；
- 避免无意义的专有名词和强行造词。

### 17.2 英文

- 使用自然、简洁的技术英语；
- 标题可使用 Title Case，正文使用 sentence case；
- 操作名称优先使用动词加对象；
- 错误信息说明发生了什么以及下一步如何处理；
- 不使用空泛的营销最高级表达。

### 17.3 三语言一致性

- 简体中文、台湾繁体和英文表达含义一致，不要求逐字直译；
- 日期、阅读时间和无障碍标签按语言本地化；
- 三种语言的信息层级和稳定字段保持一致；
- 同一 slug 的三语言内容文件必须同时存在。

## 18. 设计禁用项

不得采用：

- 更换或误写品牌标准色；
- Apple 蓝作为全站主色；
- 大面积纯白或纯黑背景；
- 全站 Liquid Glass；
- 内容卡片大面积毛玻璃；
- 彩虹、霓虹和复杂渐变；
- 厚重阴影；
- 密集小卡片墙；
- 后台管理系统式胶囊导航；
- 自动播放轮播；
- 持续循环动画；
- 依赖 hover 才能发现的核心功能；
- 低对比度细字体；
- 无标签图标按钮；
- 把多个项目放在同一个详情页；
- 为了模仿 Apple 而复制系统控件和页面结构。

## 21. 当前实现与规范的差异

本文件是目标设计基线。当前代码总体方向一致，但后续实施时应优先处理以下差异：

1. 当前部分间距仍存在 17px、27px、42px、54px 等值，应逐步映射到 4px 间距体系；
2. 当前焦点主要使用单层 outline，可根据背景情况升级为两层焦点环；
3. 应补充 200% 缩放和大字体情况下的专项验收；
4. View Transitions API 不可用时只能降级为淡出淡入，应持续关注浏览器支持范围；
5. 200×200等超大图纸会显著增加项目详情页传输量，后续可在不删除原图链接的前提下评估响应式缩略图。

处理这些差异时应采用小步修改，不同时重构内容结构和视觉系统。

## 22. 规范维护

- 本文件记录设计决策和行为标准；
- `src/styles/tokens.css` 是运行时代码中的颜色与基础令牌来源；
- 组件不得私自新增重复颜色、字体或动效常量；
- 新增全局设计令牌时，先更新本规范，再更新代码；
- 仅影响单个内容页面的特殊排版不应上升为全局令牌；
- 设计规范发生实质变化时更新版本和日期；
- 所有修改仍以内容可读性、可访问性和长期维护成本为最终判断标准。
